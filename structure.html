Structure & Layout of the Extension

This page explains the architecture of the Academic-CV Template extension.

1. Data Caching Mechanism

To optimize performance and respect Google API rate limits, the extension implements a configurable caching strategy.

  • Cache File: Fetched data is serialized into a hidden JSON file named .cv_data.json located in your project root. This file serves as the data source for all Lua shortcodes during rendering.
  • Configuration: The caching behavior is controlled via the google-document section in your _quarto.yml. You can adjust the validity period (cache-update-interval), disable updates entirely for offline work (cache-update: false), or disable caching to always fetch fresh data (sheet-cache: false).
  • Default Behavior: By default, the cache is considered valid for 24 hours. If the file exists and is younger than this interval, the extension skips the API fetch.
  • Forcing Updates: You can force a fresh fetch by either:
    1. Setting sheet-cache: false temporarily in your YAML.
    2. Manually deleting the .cv_data.json file.

2. Extension Components

The extension operates as a Quarto Project Extension using several interacting components:

Pre-render Script

Files: _fetch_cv_data.R

Location: acvt/R/

The script _fetch_cv_data.R runs before any content in your .qmd file is processed. It handles authentication, fetches data from Google Sheets, and writes the structured data to .cv_data.json in the project root folder.

Filters

Files: typst-inject-metadata.lua, typst-cover-letter.lua, typst-embed-attachments.lua, typst-layout.lua, typst-styling.lua

Location: acvt/filters/

Lua filters intervene in the Pandoc conversion process to inject data and handle special content.

  • typst-inject-metadata.lua: Reads the entire YAML data and converts it into Typst variables. The Variables are then passed to the typst template.
  • typst-cover-letter.lua: Extracts the content under ## Coverletter and makes it available to the template.
  • typst-embed-attachments.lua: Handles the inclusion of appendix documents.
  • typst-layout.lua: This filter acts as an adapter that transforms the format-agnostic AST grid structure generated by shortcodes into the native Typst function calls required by the template.
  • typst-styling.lua: Transforms custom Pandoc Span attributes into Typst #text() function calls.

Shortcodes

Location: acvt/shortcodes/

The Lua scripts in this directory provide the user-facing commands used in the .qmd file. To ensure maintainability and extensibility, the code is organized into a modular architecture:

  1. Engine (engine-*.lua): Handles data retrieval, parsing, and logic. Format-agnostic.
  2. Renderer (renderer-*.lua): Orchestrates the process and handles error checking.
  3. Strategies (typst-*.lua): Defines how the data is visually constructed for the specific format.
  4. Entry Points: The actual files wired to the shortcode in _extension.yml.

1. Shared Utilities

  • utils.lua: Contains central helper functions for string manipulation, file I/O, and type checking used by all other modules.

2. CV Sections (cvs/)

Handles the standard grid layout for CV entries and research interests.

  • Shortcodes: {{< cv-section >}}, {{< research-interests >}}
  • Core: engine-cvs.lua (Data Logic), renderer-cvs.lua (Factory).
  • Strategy: typst-cvs.lua (Generates AST Grid for the Typst Adapter).
  • Entry Points:
    • typst-cvsection.lua: Wires the standard CV section logic.
    • typst-interests.lua: Wires the research interests logic (injecting specific function names).

3. Publication Lists (pbl/)

Handles the processing of BibTeX/YAML bibliographies via Pandoc Citeproc.

  • Shortcode: {{< publication-list >}}
  • Core: engine-pbl.lua (Pandoc Pipeline & Sorting), renderer-pbl.lua (Factory).
  • Strategy: typst-pbl.lua (Generates AST Grid for the Typst Adapter).
  • Entry Point: typst-publist.lua.

4. Skill Visualization (vsl/)

Handles the visual representation of skills.

  • Shortcode: {{< visualize-skills >}}
  • Core: engine-vsl.lua (Data Logic), renderer-vsl.lua (Factory).
  • Strategy: typst-vsl.lua (Transforms data into a Typst array for the template function).
  • Entry Point: typst-skills.lua.

5. Extending the Template (Example: HTML)

The architecture is designed to support additional formats without modifying the core logic. For example, to add HTML support for CV sections, a developer would only need to add two files:

  1. The Strategy (html-cvs.lua): A Lua module that implements the renderer interface. Instead of generating Typst code or AST for adapters, it would generate native Pandoc Divs with CSS classes (e.g., for Bootstrap Grid). lua function HtmlStrategy.create_row(func_name, args, grid_config) -- Return pandoc.Div with HTML classes end
  2. The Entry Point (html-cvsection.lua): A wiring file that injects this new strategy into the existing renderer. lua local Renderer = require("renderer-cvs") local HtmlStrategy = require("html-cvs") return { ["cv-section"] = Renderer.create_shortcode_handler(HtmlStrategy) } Finally, this new entry point would be registered in _extension.yml under contributes > formats > html > shortcodes.

Template & Partials

Files: template.typ, template-typst.typ, theme.typ, utils.typ, letter.typ, cv-parts.typ, skills.typ, page.typ, typst-show.typ

Location: acvt/typst/

The main file, template.typ, orchestrates the rendering process. It determines the sequence in which the template partials are loaded and processed.

Caution

You should generally not modify template.typ directly. Instead, modify the specific partials it loads:

Partial File Purpose
typst-template.typ Main entry point logic. Orchestrates the switch between Cover Letter and CV modes.
theme.typ The Design System. Defines the color palette, fonts, sizes, and global text styles.
utils.typ Utilities. Low-level helper functions for data manipulation, string sanitization, and icon rendering.
letter.typ Cover Letter. Contains the specific layout and rendering logic for the Cover Letter.
cv-parts.typ CV Components. Contains core CV components functions: Title Page, Footer, Resume Entries, and Publication Lists.
skills.typ Skills. Specialized logic for rendering skill bars and skill tables.
page.typ Basic Page Layout. Configures global page dimensions, margins, and the page header.
typst-show.typ Show Rules. Defines Typst show rules to style raw Markdown elements (Headings, Blockquotes).

Output: By default, the generated document is saved to the _output/ directory.

3. Layout Logic (The Grid)

The template uses a consistent two-column grid layout:

  • Left Column (Sidebar): Narrow, right-aligned.
  • Right Column (Content): Wide, left-aligned.
Sidebar Column Content Column
Cell 1 Cell 2
Cell 3 Cell 4
Cell 5 Cell 6
Cell … …

Data Processing Sequence

When you use {{< cv-section >}}, the extension iterates through your data sheet row by row. For each row, it calls the specified Typst function (e.g., #resume-entry).

Important

The cv-section shortcode is agnostic regarding layout. It simply passes the column values to the function. The layout is determined entirely by the Typst function, which maps the incoming content columns to grid cells sequentially from left to right.

Example: resume-entry Mapping

The resume-entry function fills the grid cells sequentially using the columns it receives, in the following order:

  1. Column 1: Maps to Sidebar Column: Cell 1 (e.g., Date range, accent color text style).
  2. Column 2: Maps to Content Column: Cell 2 (e.g., Job Title, bolded text style).
  3. Column 3: Maps to Sidebar Column: Cell 3 (e.g., Location, oblique text style).
  4. Column 4: Maps to Content Column: Cell 4 (e.g., Description text, normal text style).
  5. Column 5: Maps to Sidebar Column: Cell 5 (e.g., Sub-label or empty).
  6. Column 6: Maps to Sidebar Column: Cell 6 (e.g., Bullet points/Details).

Without any manual ordering via a shortcode argument, the Typst function maps the content of the columns sequentially, one after the other. The first column content goes to the left column (Cell 1), the next to the right column (Cell 2), the third back to the left column (Cell 3—below the first entry), and so on.

Therefore, to fully automate document creation, the content in your Google Sheets should ideally be placed in the exact sequence you want it displayed in the grid.

However, relying solely on the spreadsheet order requires a clear vision of how you want to populate every section. To alter the sequence of the columns dynamically, you can use the column-order argument in the shortcode. This allows you to manually define the position of each column directly in the shortcode call, ensuring that your “Date” column is passed as Column 1, your “Title” as Column 2, and so on, regardless of their order in the source sheet.

4. Typst Function Reference

The layout functions are defined in typst/cv-parts.typ (title page, footer, resume entries, publications list) and typst/skills.typ (skill bar). By allowing you to specify the func in the shortcode, the template provides the additional flexibility to choose different layouts for different sections of your CV (e.g., a detailed list for jobs vs. a simple list for interests) without changing the underlying data structure. This extension comes with the following Typst functions to display different content types.

Function Layout Behavior Use Case Shortcode
resume-entry Maps arguments to the 2-column grid (Left, Right, Left, Right…). Standard CV entries (Experience, Education). {{< cv-section >}}
research-interests Skips the first few grid positions to place content directly in the right column. Simple lists or bullet points. {{< research-interests >}}
publication-list Renders a formatted bibliography list. Publication sections. {{< publication-list >}}
visualize-skills-list Renders a graphical bar chart table. Visualizing skill levels. {{< visualize-skills >}}

You can also write and provide your own custom Typst functions. To do this, declare a new function in typst/cv-parts.typ and then call it via the func argument in the shortcode (e.g., {{< cv-section func="new_function_name" >}}).

See Shortcodes & Content for usage examples.

5. Package Architecture & Maintenance

The acvt R package acts as a container and manager for the Quarto extension. It ensures that users can easily create, configure, and update their CV projects.

The “Source of Truth”

The master version of the template files (QMD, YAML, CSS, Lua) is stored inside the R package structure at: inst/rstudio/templates/project/skeleton/

Update Workflows

The package provides three distinct mechanisms to handle template updates, depending on the target:

  1. Project Update (update_current_cv_project):
    • Source: The installed R package library.
    • Target: The _extensions/ folder of your current project.
    • Use Case: You updated the R package and want to apply fixes to an existing CV.
  2. Template Hotfix (update_template_from_git):
    • Source: The GitHub repository (ZIP download).
    • Target: The installed R package library.
    • Use Case: You want to fetch the latest template version for new projects without reinstalling the full package.
  3. Developer Sync (dev_sync_template):
    • Source: Your local source code folder (inst/).
    • Target: The installed R package library.
    • Use Case: For package developers to test changes in the “New Project” wizard immediately.
